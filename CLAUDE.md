## Layer Contract

- **Routes** declare endpoints and apply `authenticate` middleware. They do not contain logic.
- **Controllers** parse and validate the request (Zod via the validator), call exactly one repository method per operation, and return the response. They do not query the database.
- **Repositories** own all Prisma calls. They do not validate input or check ownership.
- **Validators** expose typed `validate<Entity>Create` / `validate<Entity>Update` functions backed by Zod schemas.

## Security Invariants

These are enforced by the generated code and must not be removed:

1. **Integer ID validation** вЂ” `_parseId` in every controller rejects floats, alpha strings, SQL-injection suffixes, and overflow values. Never bypass it.
2. **Owner FK injection** вЂ” the `userId` FK is always set from `req.user.id` inside the controller. It is never read from the request body. The Zod validator excludes it from all schemas.
3. **Ownership check before mutation** вЂ” `update` and `remove` fetch the existing record and compare its owner FK against `req.user.id` before proceeding. Do not remove these checks.
4. **Auth entity self-ownership** вЂ” users may only update/delete their own `User` record; the controller enforces `req.user.id === parseInt(req.params.id)`.
5. **Sensitive fields hashed** вЂ” fields marked `@llm sensitive` in the schema are hashed with bcrypt before storage and excluded from all API responses and the JWT payload.
6. **JWT required on writes** вЂ” `authenticate` middleware is applied to all `POST`, `PUT`, `PATCH`, and `DELETE` routes. Reads may be public depending on the schema.
## Adding a Feature

Follow the layer contract strictly:

1. Add the route declaration in `src/routes/<entity>.routes.ts`
2. Add the handler in `src/controllers/<entity>.controller.ts` вЂ” validate input with `validate<Entity>Create/Update`, inject owner FK from `req.user.id`, call the repository
3. Add the data-access method in `src/repositories/<entity>.repository.ts`
4. If new fields are needed, update `src/validators/<entity>.validator.ts` (Zod schemas) and `src/types/<entity>.types.ts`

Never put Prisma calls in a controller. Never put validation logic in a repository.

## Error Handling

Throw typed errors from `src/lib/errors.ts`; the global handler in `src/app.ts` maps them to HTTP status codes:

| Class | Status |

### Non-Negotiable Rules

**Before every apply:**
- Run `terraform plan` first. Read every line. Never skip this step.
- Flag these keywords вЂ” they require explicit human approval before applying:
  - `forces replacement` вЂ” the resource will be destroyed and recreated (potential data loss)
  - `destroy` вЂ” the resource will be permanently deleted
  - `known after apply` вЂ” the value is computed at apply time; verify downstream impact

**State is sacred вЂ” never:**
- Manually edit `.tfstate` or `.tfstate.backup`
- Delete the remote state backend (S3 bucket, GCS bucket) while resources exist
- Run two concurrent `terraform apply` operations against the same workspace
- If state is corrupted: stop immediately, do not apply, investigate first

**Secrets вЂ” never hardcode:**
- `db_password`, API keys, and all sensitive variables must be supplied via:
  - `TF_VAR_db_password=...` environment variable, or
  - `terraform apply -var="db_password=..."` CLI flag, or
  - A `terraform.tfvars` file that is in `.gitignore`
- Never commit `.tfvars` files that contain secrets to version control

**Destructive operations require explicit human confirmation:**
- `terraform destroy` вЂ” tears down all managed infrastructure
- `terraform apply` when the plan contains any `destroy` actions
- `terraform state rm` вЂ” removes a resource from state (does not destroy it in the cloud)
- `terraform taint` / `terraform apply -replace` вЂ” forces resource recreation

**Modifying existing resources:**
- Adding a resource: safe after plan review
- Modifying a resource: check whether the change `forces replacement`
- Renaming a resource block in HCL: use `terraform state mv old new` вЂ” never delete and recreate
- Removing a resource from HCL: it will be destroyed on next apply вЂ” confirm this is intended

**Backend and state migration:**
- Never edit `backend.tf` without immediately running `terraform init -migrate-state`
- Confirm the migration succeeded before removing the old state location

**Lock file:**
- Always commit `.terraform.lock.hcl`; never edit it manually
- To upgrade providers: `terraform init -upgrade`, review the diff, commit the result

**Drift detection:**
- Run `terraform plan` at the start of every infrastructure change session
- If plan shows changes you did not make, investigate drift before applying
